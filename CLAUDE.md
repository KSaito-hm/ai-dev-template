## セッション開始チェックリスト

セッションを開始したら、必ず以下の順で確認すること：

1. `docs/00_context.md` を読む（プロジェクト背景・目的の把握）
2. `docs/04_tasks.md` を読む（現在のフォーカスタスクと進捗の確認）
3. `git branch` と `git log --oneline -5` で現在の状態を確認する
4. 人間に現状サマリーと次のアクション候補を提案する

---

## 絶対ルール

### ① ドキュメント優先原則

**実装より先にドキュメントを更新すること。この順序を逆にすることは禁止。**

変更が発生したとき、必ず以下の順で更新する：

```
01_requirements.md → 02_architecture.md → 03_design.md → 04_tasks.md → 実装
```

- チャット上だけで仕様を決定することは禁止
- ドキュメントを更新せずに実装を進めることは禁止
- 実装完了後、ドキュメントと実装の整合を必ず確認すること

### ② Git Flow 厳守

**全作業は `feature/T-xxx` ブランチで行うこと（xxx はタスクID）。**

| ブランチ操作 | ルール |
|---|---|
| feature ブランチ作成 | `git checkout -b feature/T-xxx develop` からのみ切る |
| feature ブランチへのコミット | 自由に行ってよい。メッセージは `type(T-xxx): 内容` 形式 |
| feature → develop マージ | `--no-ff` 必須。人間の確認後に実行する |
| develop → main マージ | `release/x.x.x` ブランチ経由のみ。人間の承認後に実行する |
| **develop への直接コミット** | **禁止** |
| **main への直接コミット** | **禁止** |

**ブランチ命名規則：**

| 種別 | 命名 | 例 |
|---|---|---|
| 機能・タスク | `feature/T-xxx` | `feature/T-101` |
| リリース | `release/x.x.x` | `release/v1.0.0` |
| 緊急修正 | `hotfix/T-xxx` | `hotfix/T-999` |

### ③ タスク完了チェックリスト

タスクを「完了」とする前に、必ず以下を確認すること：

- [ ] 実装が `03_design.md` の設計と整合している
- [ ] `04_tasks.md` のステータスを `完了` に更新した
- [ ] 完了タスクログに記録した（完了日・ブランチ名を記入）
- [ ] feature ブランチを develop へ `--no-ff` でマージした
- [ ] 次のアクションを人間に提案した

## フェーズ別ルール

### ドキュメントフェーズ（T-001〜T-004）

- `T-001`（00_context.md）が完了するまで他のタスクを開始しないこと
- `02_architecture.md` のステータスが「承認済み」になるまで `03_design.md` の作業を始めないこと
- `03_design.md` のステータスが「承認済み」になるまで実装を始めないこと

### 実装フェーズ（T-101〜）

- 実装前に必ず `03_design.md` の該当箇所を確認すること
- 設計と実装に乖離が生じた場合は、実装を止めてドキュメントを先に更新し、人間の確認を取ること
- 設計にないものを実装しないこと（勝手な拡張禁止）

---

## 禁止事項

- `develop` / `main` への直接コミット
- ドキュメント更新なしの実装
- チャット上だけでの仕様決定
- 人間の確認なしの `develop` / `main` へのマージ
- `03_design.md` が「承認済み」になる前の実装開始
